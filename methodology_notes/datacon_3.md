# Источник — это не просто URL-адрес
Для каждого источника необходимо знать:
- какие записи он содержит
- откуда взялись эти записи
- какие поля доступны
- являются ли данные первичными, вторичными или производными
- пересекается ли он с другими источниками
- можно ли его использовать легально и технически повторим ли парсинг

# PDF → данные
- большая часть работы в научных ML-проектах — это не обучение модели, а **преобразование PDF в структурированные данные**.
- экстракция - не копирование текста
- экстракция - реконструкция универсальной структуры данных

| Что видит человек | Что хранит PDF                     | Что нужно датасету                  |
| ----------------- | ---------------------------------- | ----------------------------------- |
| Абзац             | Текстовые фрагменты с координатами | Семантические утверждения           |
| Таблица           | Линии, слова, координаты           | Строки и столбцы                    |
| Молекула          | Изображение или векторный рисунок  | SMILES / InChI                      |
| График            | Пиксели или векторные контуры      | Числовые точки данных               |
| Схема реакции     | Графические объекты и подписи      | Реагенты, продукты, условия реакции |
> [!warning]+ Почему текст из PDF часто оказывается «сломленным»
> 
> PDF хранит не смысл документа, а набор текстовых фрагментов, координат и графических объектов. Поэтому при извлечении данных возникают искажения:
> 
> - теряются надстрочные и подстрочные индексы (`IC₅₀ → IC 50`, `10⁻⁶ → 10-6`);
>     
> - повреждаются специальные символы и греческие буквы (`α → a`, `μM → uM`);
>     
> - нарушается порядок чтения из-за двухколоночной вёрстки;
>     
> - в текст попадают колонтитулы, номера страниц и служебные элементы;
>     
> - возникают ошибки из-за особенностей шрифтов, Unicode и OCR;
>     
> - отдельные фрагменты текста могут извлекаться вне исходного контекста.
>     
> 
> В результате сырой текст из PDF редко подходит для анализа напрямую и обычно требует очистки, нормализации и восстановления структуры документа перед использованием в RAG, ML-пайплайнах или датасетах.


## Типы PDF-документов

| Тип PDF                  | Описание                                                           | Стратегия обработки                                      | Особенности                                                        |
| ------------------------ | ------------------------------------------------------------------ | -------------------------------------------------------- | ------------------------------------------------------------------ |
| Born-digital PDF         | Создан напрямую из Word, LaTeX, XML и других редакторов            | Извлечение текста и структуры документа                  | Самый простой для обработки тип PDF; текст обычно доступен без OCR |
| Publisher XML-backed PDF | PDF сгенерирован из структурированного XML-источника издателя      | Использовать XML или HTML вместо PDF при наличии доступа | Позволяет получить данные без потерь структуры и форматирования    |
| Mixed PDF                | Содержит одновременно текст, изображения и сканированные элементы  | Гибридный пайплайн                                       | Требуется определять тип контента для каждого блока страницы       |
| Form-like PDF            | Формы, анкеты, отчёты с фиксированными полями                      | Извлечение полей формы                                   | Структура документа обычно заранее известна и стабильна            |
| Supplementary PDF        | Дополнительные материалы к статье, часто содержат таблицы и данные | Извлечение таблиц в первую очередь                       | Обычно являются наиболее ценным источником данных для датасетов    |
| Scanned PDF              | Каждая страница представляет собой изображение                     | Сначала OCR, затем извлечение данных                     | Текст отсутствует как объект документа, качество зависит от OCR    |
| Patent PDF               | Патенты с большим объёмом текста, формул и технических описаний    | Извлечение разделов и сущностей                          | Содержат много шума, повторов и сложных формулировок               |

### Краткая интерпретация для ML/Data Engineering

|Приоритет обработки|Тип|
|---|---|
|🟢 Самые удобные|Born-digital PDF, XML-backed PDF|
|🟡 Средняя сложность|Mixed PDF, Form-like PDF|
|🟠 Сложные|Supplementary PDF|
|🔴 Самые сложные|Scanned PDF, Patent PDF|

Главная мысль слайда: **не существует универсального парсера PDF**. Перед извлечением данных необходимо определить тип документа и выбрать соответствующий пайплайн обработки. Один и тот же подход плохо работает одновременно для сканов, патентов и структурированных PDF, поэтому современные системы обычно начинают с этапа классификации документа.


# Пайплайн извлечения данных из PDF

1. **Определение структуры (Detect structure)**
   - Определить типы элементов на странице: текст, таблицы, изображения, графики, схемы.

2. **Парсинг объектов (Parse objects)**
   - Извлечь содержимое каждого объекта в подходящем формате.

3. **Сегментация контента (Segment content)**
   - Разбить данные на логические блоки: разделы, абзацы, таблицы, сущности.

4. **Валидация (Validate)**
   - Проверить полноту, корректность и согласованность извлечённых данных.

> Независимо от того, является PDF текстовым или сканированным изображением, общая логика извлечения остаётся одинаковой. Меняются только инструменты (парсеры, OCR, CV-модели).


Сначала нужно определить тип документа:

1. Born-digital PDF → PyMuPDF.
2. Научная статья → GROBID / Docling.
3. Таблицы → pdfplumber / Camelot.
4. Скан → OCR.
5. Сложная вёрстка → AI Layout Parser.

Только после этого выбирать инструменты извлечения данных.
## Анализ структуры PDF (Layout Analysis)

| Подход | Как работает | Инструменты | Ограничения |
|---------|-------------|-------------|-------------|
| **Text-based** | Анализирует текстовые объекты PDF: координаты, шрифты, порядок чтения | [pdfminer.six](https://github.com/pdfminer/pdfminer.six), [PyMuPDF](https://pymupdf.readthedocs.io/), [pdftotext](https://github.com/jalan/pdftotext) | Не понимает семантику документа; изображения, графики и таблицы часто теряются |
| **Image-based** | Рассматривает страницу как изображение и использует OCR/CV-модели | Tesseract, LayoutParser, PubLayNet | Зависит от качества изображения и обучающих данных; ошибки в формулах и химических обозначениях |

### Когда использовать

| Ситуация | Подход |
|-----------|---------|
| PDF из Word, LaTeX, XML | Text-based |
| Сканированный документ | Image-based |
| Научная статья | Гибридный |
| Supplementary PDF | Обычно гибридный |

|Задача / сценарий|Класс инструмента|Инструменты|Лучше всего подходит для|Ограничения|
|---|---|---|---|---|
|Простая таблица с линиями|Table extractor|Camelot (lattice)|Таблицы с видимой сеткой|Не работает без линий|
|Таблица без границ|Table extractor|Camelot (stream), pdfplumber|Таблицы, разделённые пробелами|Чувствителен к форматированию|

|Структура научной статьи|Article parser|GROBID, CERMINE, Science Parse|Заголовок, авторы, аннотация, ссылки, секции|Не извлекает сложные таблицы|
|PDF → Markdown для RAG|Conversion parser|PyMuPDF4LLM, Marker, Docling, Unstructured|LLM-ready чанки|Требуется проверка структуры|
|PDF → Markdown для RAG (облачный/AI)|AI conversion parser|Mistral OCR, LlamaParse, Marker, Docling|Быстрая подготовка документов для LLM|Markdown может скрывать ошибки извлечения|
|Сложный макет документа|AI layout parser|Docling, DeepDoctection, LayoutParser|Блоки, таблицы, изображения, регионы|Качество зависит от модели|

### Что можно извлечь из научного рисунка

После обработки рисунок превращается не просто в PNG-файл, а в структурированный объект:

| Поле | Описание |
|--------|----------|
| figure | Номер рисунка |
| page | Страница документа |
| image | Файл изображения |
| bbox | Координаты рисунка на странице |
| caption | Подпись к рисунку |
| entities_in_caption | Сущности из подписи |
| entities_in_text | Сущности из окружающего текста |
| data_relevance | Тип данных (график, спектр, схема реакции и т.д.) |

#### Пример структуры

```json
{
  "figure": "Figure 3",
  "page": 6,
  "caption": "Binding curves for aptamer A1...",
  "image": "paper_001_fig3.png",
  "entities": ["A1", "thrombin"],
  "data_relevance": "binding_curve"
}
```

# Экстракция данных из веб-страниц

**Экстракция данных (Web Data Extraction)** — процесс получения структурированных данных из веб-страниц для дальнейшего анализа или загрузки в БД.

**Типичный пайплайн:**

```text
Web page
    ↓
HTML / API response
    ↓
Извлечение данных
    ↓
Таблица записей
    ↓
Нормализация
    ↓
Готовый датасет
```
#### Что можно извлекать

**Общие данные:**

- Текстовый контент
    
- Таблицы
    
- Ссылки
    
- Метаданные
    
- Ответы API (JSON/XML)
    
- Файлы (PDF, CSV, DOCX и др.)
    
- Динамически загружаемый контент (JavaScript)
    

**Химические и научные данные:**

|Тип данных|Примеры|
|---|---|
|**Compound data**|название вещества, синонимы, химическая формула, молекулярная масса, SMILES, InChIKey|
|**Assay data**|биологическая активность, мишень (target), тип теста, значение, единицы измерения, условия эксперимента|
|**Reaction data**|реагенты, катализаторы, растворители, температура, время реакции, выход продукта|
|**Material data**|состав, метод синтеза, площадь поверхности, размер пор, стабильность|
|**Supplier metadata**|номер каталога, чистота, наличие, фасовка, цена|
|**Publication metadata**|название статьи, авторы, DOI, аннотация, дополнительные материалы, таблицы|


#### Основные инструменты
|Задача|Инструменты|Кратко|
|---|---|---|
|Загрузка страниц|requests, httpx, urllib|Получение HTML, JSON, файлов по URL|
|Парсинг HTML|BeautifulSoup, lxml, parsel, selectolax|Извлечение текста, ссылок и элементов из HTML|
|Извлечение таблиц|pandas.read_html|Автоматическое чтение HTML-таблиц в DataFrame|
|Работа с JSON и API|json, pandas.json_normalize|Обработка ответов API и преобразование JSON в таблицы|
|Автоматизация браузера|Selenium, Playwright|Работа с JavaScript-сайтами через реальный браузер|
|Массовый веб-скрейпинг (краулинг)|Scrapy|Сбор данных с большого количества страниц|
|Валидация данных|pydantic, pandera|Проверка структуры, типов и качества данных|
|Хранение данных|csv, jsonl, parquet, sqlite, duckdb|Сохранение результатов в файлы и БД|


После извлечения данные обычно приводят к единому формату:

```text
Название вещества
    ↓
SMILES / InChIKey
    ↓
Нормализация единиц измерения
    ↓
Удаление дубликатов
    ↓
Готовый химический датасет
```

**Результат:** структурированные данные (CSV, JSON, DataFrame, SQL, Vector DB), пригодные для аналитики, ML, RAG и построения химических баз знаний.
