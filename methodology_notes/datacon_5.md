# RAG


- rag может быть использован несколько раз под многокомпонентные запросы
- 
  
1. **агент поисковик** - находит в статье нужные фрагменты
   - убирает мусорные слова
1. **агент по экстракции данных** - заполняет поля в таблице
- **Исходные данные:** абзац из научной статьи.

  > «Наночастицы серебра (Ag NP) диаметром 20 нм показали цитотоксичность IC₅₀ = 12 мкг/мл в отношении клеток линии HepG2 после 24 ч инкубации...»

- **Результат экстракции:** структурированная запись.

| Поле       | Значение  |
| ---------- | --------- |
| Материал   | Ag NP     |
| Размер     | 20 нм     |
| Показатель | IC₅₀      |
| Значение   | 12 мкг/мл |
| Клетки     | HepG2     |
| Время      | 24 ч      |

> **Цель:** автоматически получать структурированные записи из тысяч научных статей с возможностью проверить источник каждого извлечённого поля.


1. **агент валидатор**
- проверка единиц измерения, диапазоны, источник

**Риски**
- Плохой поиск → плохой ответ. Качество retrieval определяет результат.
- Правильная нарезка на чанки критична: слишком большие — много лишнего, слишком маленькие — теряется контекст.
	- подбирается опытным путем.
- Качество базы знаний влияет на качество ответов (GIGO: *Garbage In, Garbage Out*).
- Ошибки в RAG сложно диагностировать: проблема может быть на любом этапе — от индексации и эмбеддингов до поиска и генерации.
